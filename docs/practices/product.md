# Product Practices

## Core Principle

**Simplicity first**: Every product decision prioritizes simplicity for end users and maintainability for the team.

**Startup agility**: Ship fast, learn from users, iterate quickly. Optimize features and workflows only when usage data shows it's needed.

## Product Development Process

### 1. Product Discovery
- Start with user problems, not solutions
- Validate assumptions with minimal effort
- Focus on core value proposition before adding features
- Ask: "What's the simplest way to solve this problem?"

### 2. Product Definition
- Define clear, measurable success criteria
- Prioritize features that deliver maximum value with minimum complexity
- Document user journeys in simple, clear language
- Establish what we won't build (avoid scope creep)

### 3. Product Development
- Build incrementally: smallest viable feature first
- Test with real users early and often
- Remove features that don't add clear value
- Maintain focus: one clear goal per release
- Ship fast, learn fast - don't wait for perfection
- Optimize features based on actual usage, not assumptions

### 4. Product Launch
- Launch with minimal viable feature set
- Provide clear, simple documentation for users
- Monitor core metrics that matter
- Plan for quick iteration based on feedback

### 5. Product Maintenance
- Regularly review and remove unused features
- Simplify workflows based on user feedback
- Keep documentation current and accessible
- Maintain backward compatibility when possible

### 6. Product Retrospective
- Measure against original simplicity goals
- Identify complexity that crept in
- Document learnings for future decisions
- Celebrate simplicity wins

### 7. Product Iteration
- Iterate based on user needs, not feature requests
- Prefer improving existing features over adding new ones
- Remove complexity before adding new capabilities
- Maintain clear product vision

### 8. Product Optimization
- Optimize for user time and cognitive load
- Reduce steps required to complete core tasks
- Eliminate unnecessary options and choices
- Measure success by simplicity metrics

## Usability Metrics

### 1. Core Usability Metrics
- **Task Success Rate**: Percentage of users who complete key tasks
- **Time to Complete**: How long users take to accomplish goals
- **Error Rate**: Frequency of user errors and recovery time
- **Learnability**: How quickly new users become productive
- **Satisfaction**: User satisfaction scores (NPS, CSAT, SUS)

### 2. Simplicity Metrics
- **Steps to Complete**: Count of steps for core user journeys
- **Cognitive Load**: Measure of mental effort required
- **Feature Discoverability**: How easily users find features
- **Documentation Dependency**: How often users need help/docs
- **Configuration Complexity**: Number of settings/options required

### 3. Measuring and Tracking
- Establish baseline metrics before changes
- Track metrics continuously, not just at launch
- Use both quantitative (analytics) and qualitative (user feedback) data
- Focus on metrics that indicate simplicity success
- Review metrics regularly in product retrospectives

### 4. Metric-Driven Decisions
- Use metrics to identify simplification opportunities
- Prioritize improvements based on metric impact
- Celebrate when metrics show increased simplicity
- Remove features that don't improve core metrics
- Optimize workflows only when metrics show problems
- Let usage data drive optimization, not assumptions

## When to Optimize

### Optimization Triggers
- **Usage Patterns**: When metrics show users struggling with workflows
- **Scale**: When user growth creates actual bottlenecks
- **Feedback**: When consistent user feedback indicates problems
- **Business Metrics**: When core metrics (conversion, retention) are impacted

### Optimization Approach
1. **Measure First**: Use analytics and user feedback to identify real problems
2. **Prioritize by Impact**: Focus on optimizations that affect most users or core metrics
3. **Test Changes**: A/B test optimizations when possible
4. **Iterate Quickly**: Make small improvements and measure results
5. **Remove What Doesn't Work**: Don't be afraid to revert or remove features

### What NOT to Optimize
- Don't optimize workflows that users aren't complaining about
- Don't add features to solve problems that don't exist
- Don't optimize for edge cases before common cases
- Don't build enterprise features without enterprise needs

## Decision Framework

When evaluating any product decision, ask:
1. Does this make it simpler for the end user?
2. Does this make it simpler to maintain?
3. Can we achieve the same outcome with less complexity?
4. What are we willing to remove to add this?

## Anti-Patterns to Avoid

- Adding features "just in case"
- Building for hypothetical future needs
- Complex configuration when defaults would suffice
- Multiple ways to accomplish the same task
- Features that require extensive documentation to use
- Optimizing before measuring user behavior
- Building enterprise features without enterprise scale
- Waiting for perfection before shipping
